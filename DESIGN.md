## 一、需求梳理

### 1.1 核心目标

构建一个"AI 智囊团"——不只是更强的模型，而是有行业认知深度的专家集合。

| 维度 | 描述 |
|------|------|
| **核心目标** | 多领域顶级思考者蒸馏成 Skill，按需调用 |
| **调用方式** | 自动路由 or 手动指定专家 |
| **架构形式** | 独立 Skill，可被 jarvis-expert 引用 |
| **可扩展性** | 复用社区已蒸馏人物 + 自蒸馏新专家 |

### 1.2 命名：为什么叫「问天」

- **原型**：屈原《天问》，向天地万物发出 172 个根本性追问，涵盖宇宙、历史、人生
- **意象**：向各领域最高智者发问，按问题性质召唤对应的"知者"
- **对齐**：女娲（造人）→ 仓颉（造字）→ 问天（问道），同属中国神话/文化语境

### 1.3 架构选型：为什么是独立 Skill

- **关注点分离**：jarvis-expert 核心职责是"高质量决策建议"，专家蒸馏是独立能力层
- **复用性**：主 Jarvis 和 jarvis-expert 均可直接调用
- **可维护性**：新增/更新专家只改问天目录，不影响其他组件
- **集成方式**：jarvis-expert SKILL.md 末尾加一行引用即可赋能

### 2.2 女娲 vs 仓颉

| 维度 | 女娲 | 仓颉 |
|------|------|------|
| **核心理念** | 角色扮演 + 思维框架 | 认知植入（Harness Engineering）|

| **已蒸馏人物** | 13 个 | 3 个（质量更高） |

### 2.3 选型结论

| 场景 | 推荐 |
|------|------|
| 快速复用已有人物 | **女娲** 13 人物直接拿 |
| 高质量蒸馏新人物 | **仓颉**（质量更高，有自检工具） |
| Skill 持续优化 | **达尔文** |
| 蒸馏内部人员 | **同事.skill** |

## 三、系统架构

### 3.1 整体架构

```
用户问题
    │
    ▼
┌─────────────────────────────────────┐
│            问天.skill                │
│         （智囊团 · 路由层）           │
│                                     │
│  ┌──────────────────────────────┐   │
│  │      专家路由器（Router）      │   │
│  │  问题分类 → 匹配专家 → 激活   │   │
│  └──────────────┬───────────────┘   │
│                 │                   │
│  ┌──────────────▼───────────────┐   │
│  │         专家注册表            │   │
│  │  id / 领域 / 触发词 / 来源    │   │
│  └──────────────────────────────┘   │
└──────┬──────────┬──────────┬────────┘
       │          │          │
       ▼          ▼          ▼
  [乔布斯]    [芒格]    [虎山行不行]  ...
  产品/设计  投资/决策   人性/职场
```

### 3.2 三级路由逻辑

1. **明确指定**：「用芒格的方式帮我分析」→ 直接激活
2. **关键词匹配**：检测 trigger_keywords → 建议对应专家
3. **LLM 分类**：无法匹配时，轻量分类 prompt → 路由

**多专家联合模式**：跨域问题可激活 2-3 个专家，分别给视角后整合呈现（"圆桌会议"）

### 3.3 专家注册表字段（expert-registry.md）

```yaml
id: steve-jobs
name: 乔布斯
source_skill: alchaincyf/steve-jobs-skill
distill_engine: nuwa  # nuwa / cangjie / custom
domain:
  - 产品设计
  - 用户体验
  - 品牌与发布
trigger_keywords:
  - 产品决策
  - 功能取舍
  - 简洁
  - 极致
avoid:
  - 投资
  - 大组织管理
quote: "Focus means saying no to a hundred good ideas."
```

## 六、主 SKILL.md 核心逻辑（草案）

```markdown
# 问天.skill

> 遂古之初，谁传道之？——向各领域最高智者发问。

## 触发场景
面对决策、分析、建议、创意类问题时激活。

## 专家路由
1. 读取 expert-registry.md，获取专家清单
2. 分析问题，匹配触发关键词或 LLM 分类领域
3. 激活 1-3 个最相关专家的 SKILL.md
4. 以对应专家的认知框架思考并回答
5. 多专家时：整合不同视角，标注来源，统一呈现

## 调用语法
  "用[专家名]的方式帮我看看..." → 直接激活
  "问一下智囊团..."             → 自动路由
  "几个专家一起来分析..."       → 联合模式（圆桌）
```
